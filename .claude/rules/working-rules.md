# Project Working Rules

本文件记录本项目的协作与方案输出规则。除非用户明确覆盖，后续在本项目中工作时必须遵守。

## 1. 禁止自行推断 / 猜测 / 脑补

- 不得在需求不明确、数据缺失、业务含义不确定时自行补全关键事实。
- 不得把 mock 数据、示例字段、视觉细节、业务规则当作真实要求直接落地。
- 不得用“看起来应该是”“一般会这样”替代用户确认。
- 任何会影响结构、交互、业务语义、数据模型、字段含义、页面范围、删除/覆盖行为的决定，都必须基于：
  1. 用户明确说明；或
  2. 仓库现有代码/文档中可验证的事实；或
  3. 先向用户提问得到确认。

## 2. 必要时必须提问

以下情况必须先提问，不要直接执行：

- 存在多个合理实现路径，且选择会影响用户体验、数据结构、页面范围或后续维护成本。
- 用户只描述目标，没有说明具体范围，例如“history 页面”但仓库存在多个 `prototype/**/history.html`。
- 需求涉及业务语义，例如状态成功/失败、失败通道、外部信息 map、tag 颜色含义等。
- 需求涉及删除、批量删除、覆盖、迁移、注册新页面、修改导航或构建配置。
- 无法确认是只改一个页面、全部同类页面，还是只改当前设备页面。

提问时应给出推荐项，并说明为什么推荐；不要问“是否继续执行方案”这类空泛问题。

## 3. 方案必须至少提供 3 个选项

当用户要求“分析 / 方案 / 怎么设计 / 怎么展示 / 怎么实现 / 选型 / 重构 / 优化方向”时，必须至少提供 **3 个可选方案**。

每个方案至少包含：

- 方案名称
- 核心做法
- 收益
- 风险点
- 适用场景
- 不适用场景或代价

如果用户已经指定唯一实现方式，则可以不列 3 个方案，但仍需指出关键风险与收益。

## 4. 风险点与收益必须显式列出

所有非平凡方案必须显式列出：

- 收益：用户价值、维护性、可扩展性、实现成本优势等。
- 风险点：UI 信息密度、交互复杂度、数据结构不稳定、跨文件一致性、回退难度等。
- 工时估算：用人·天或小时表达。
- 回退成本：说明撤销该方案大概需要多少工作量。

如方案影响超过 1 个文件、预计超过 1 小时、或涉及架构/交互选择，还必须遵守全局 `solution-roi-scorer` 要求：提供 ROI 评分、三分类风险评估、8 节结构方案。

## 5. 原型页面特别规则

- 涉及 `prototype/**/*.html` 的创建、修改、重构，必须委托 `scada-hmi-builder` agent 执行。
- 在委托 agent 前，如果需求存在范围或业务语义不确定，必须先向用户提问。
- Agent prompt 必须包含用户已确认的范围、列结构、交互细节、视觉约束和不可改动项，避免 agent 自行脑补。

## 6. 输出风格

- 默认用中文说明。
- 结论先行，推荐方案放在最前。
- 不输出没有依据的断言；如果是推测，必须标注“假设”并说明需要用户确认。
- 对多方案比较，优先使用表格呈现，便于用户决策。

## 7. 最小改动原则 (Minimal Change)

所有代码修改、执行计划、agent 产出的调整方案，必须遵守最小改动原则。

### 规则

- **局部修补优先**：改一个文件、一个 CSS 类能解决的问题，不做多文件全局替换。
- **改动半径 ≤3 文件**：超过 3 个文件或多于 50 行的方案，必须先在方案里显式说明"为什么局部修补不可行，为什么必须跨文件"。
- **扩展现有结构**：新增功能优先在现有页面 DOM/CSS/JS 架构上扩展（加一个 `<div>`、一条 CSS 规则、一个事件处理函数），不推翻重建页面结构。
- **聚焦用户要求**：改动严格聚焦用户指定的功能/修复/调整，不顺手"优化"无关区域、不改其他页面的相同模式（除非用户明确要求全量统一）。
- **≤5 文件或 ≤200 行的方案可执行**：超过此阈值的方案，必须先产出方案文档（含改动范围矩阵）并获得用户确认后再执行。
- **独立可回滚**：每批改动独立 commit。改一组、验证、commit，再改下一组。

### 适用范围

- 主会话中我（主 agent）的直接代码修改
- `pm` agent 产出的调整方案（必须标注受影响文件数、预估改动行数）
- `prototype-ui-reviewer` agent 的视觉实现（优先改目标文件的对应 CSS 类，不全局 sed 全项目）
- 所有 batch/bulk 操作（如 20 文件全局 token 替换）必须先获用户确认

### 为什么需要

- 项目是 20 个自包含 HTML 的纯静态原型，改一个 color token 在 20 个文件里会产生 ~900 行 diff，风险放大 20 倍
- 无自动化测试，验证依赖浏览器手动点检——改动范围越大，漏检概率越高
- GitHub Pages 直接部署 master，没有 staging 环境可以回滚
