## 1. 产品定位与概述

### 1.1 产品定位

| 属性 | 说明 |
|-----|------|
| **产品名** | SIDU-LATS（Lightweight Audit Trail System，日志审计存证系统） |
| **Phase 2 定位** | 面向 FDA 21 CFR Part 11 合规的 B/S 架构审计追踪与电子签名系统 |
| **Phase 1 定位** | 轻量级本地 exe（Windows 单机，Python + 热敏打印） |
| **核心演进** | 从"轻量合规存证工具"升级为"企业级 FDA 合规审计平台" |

### 1.2 与竞品/关联产品的定位差异

| 维度 | SIDU-LATS Phase 2 | SIDU-DHF-Hub |
|-----|-------------------|--------------|
| **架构** | B/S（Vue 前端 + Java/C# 后端） | B/S（Web 全栈） |
| **部署规模** | 单台设备 / 单用户终端 | 企业多用户服务器 |
| **数据库** | SQLite + JSONL 哈希链 | PostgreSQL |
| **核心能力** | 审计追踪 + 电子签名 + WORM 存证 | DHF 文档全生命周期管理 |
| **合规目标** | FDA 21 CFR Part 11（电子记录/签名/审计） | FDA 510(k)/DHF 文档管理 |
| **目标用户** | NTA 操作员 / QA 审计员 / 设备管理员 | 设备厂商工程师 / 质量部 |
| **硬件要求** | Windows 10 IoT 工控机（x86_64） | 独立服务器 |

### 1.3 Phase 2 核心价值

| 核心价值 | 具体描述 |
|---------|---------|
| **FDA 合规** | 完整满足 FDA 21 CFR Part 11 电子记录完整性、审计追踪、电子签名强制性要求 |
| **WORM 存证** | 分层 WORM 不可变存储（在线 VSS 快照 + 离线蓝光归档），哈希链实时固化 |
| **三权分立** | 系统管理员 / 审核人员 / QA 审计员 / 普通操作员四角色，技术层硬隔离 |
| **电子签名** | 签名人ID + 毫秒时间戳 + 签署意图三层绑定，不可剥离，不可伪造 |
| **跨平台支持** | Windows 工控机（C# .NET 8）与 Linux ARM 设备（Java Spring Boot）双后端 |
| **审计透明** | 软件自建账户，应用层直接记录 user_id，FDA 核查路径最短 |

### 1.4 目标用户与角色定义

| 角色 | 职责边界 | 可访问范围 |
|-----|---------|----------|
| **系统管理员** | 系统配置、设备对接、用户账号管理、备份操作 | 仅系统配置与运维，无检测报告查看/审批/签名权限 |
| **审核人员** | 检测数据审核、电子签名（初审/复核/批准/作废） | 仅检测报告审核，无系统配置/账号管理权限 |
| **QA 审计员** | 审计追踪复核、存证介质校验、合规审计 | 仅审计追踪查看，无数据修改/审批权限 |
| **普通操作员** | 基础数据录入、自身业务范围内数据查看 | 仅录入操作，无审批/导出/配置权限 |

> ⚠️ **三权分立强制约束**：以上四角色权限由系统层硬隔离，禁止一人兼任多岗。违反将触发越权告警。

## 2. Phase 1 → Phase 2 升级说明

### 2.1 核心变更对比

| 维度 | Phase 1（V1/V2） | Phase 2（V3） |
|-----|-----------------|---------------|
| **技术架构** | Python 单机 exe | B/S 分离架构（Vue3 前端 + Java/C# 后端） |
| **前端** | PyInstaller GUI | Vue 3 + Ant Design Vue 4 + Pinia + Vite |
| **后端** | 纯 Python | Java Spring Boot 3.2+ **或** C# .NET 8 |
| **账户体系** | Windows 系统账户集成 | **软件自建账户**（独立于 OS，与 FDA 审计透明） |
| **数据存储** | JSON 文件 | SQLite（结构化数据）+ JSONL（哈希链） |
| **密码算法** | SHA-256（hashlib） | SHA-256 + ECDSA P-256 / SM2（国密） |
| **WORM 存证** | 热敏打印机小票为主 | **分层 WORM 存储**（VSS 快照 + 哈希链 + 离线蓝光归档） |
| **权限体系** | 三角色（管理员/操作员/审计员） | **四角色（三权分立）** |
| **电子签名** | 绑定 Windows 账户 + 密码二次验证 | **签名语义绑定**（user_id + 毫秒时间戳 + 签署意图 + 私钥签名） |
| **部署方式** | Windows Service 单机 | Windows Service（工控机）/ Docker（ARM 板） |

### 2.2 升级原因说明

**1. 为什么废弃 Windows 账户体系？**

Phase 1 使用 Windows 系统账户集成，FDA 核查时存在以下问题：

| 问题 | 说明 |
|-----|------|
| **审计透明度不足** | Windows SID（S-1-5-21-xxx）对 FDA 核查员不可读，需额外映射表还原自然人身份，增加被挑战风险 |
| **运维复杂度高** | 临时账号到期自动停用、密码策略强制执行、删除账号审批均需 OS 管理员权限 |
| **双因素认证耦合** | UKey/OTP 与 Windows 账户集成需 AD 域 + RADIUS，部署成本倍增 |
| **多设备部署困难** | 每台工控机单独创建 Windows 账户，账户生命周期管理无法统一 |

**结论**：软件自建账户完全满足 FDA Part 11 要求（签名绑定自然人或机构），同时运维复杂度大幅降低。

**2. 为什么从 Python 迁移到 Java/C#？**

| 原因 | 说明 |
|-----|------|
| **FDA 合规生态** | C# .NET 8 原生支持 Windows CNG/TPM 2.0，电子签名与密钥存储更安全可靠 |
| **内存占用** | .NET Runtime ~50-150MB，Python+JVM ~200-512MB，工控机资源有限 |
| **启动速度** | .NET 冷启动快于 JVM，适合工控机环境 |
| **B/S 架构需要** | Phase 2 要求 Vue 前端 + REST API，纯 Python 无法优雅实现 |
| **维护成本** | C# .NET 8 在 Windows 工控机上运维比 Python 环境更简单稳定 |

## 3. 技术架构

### 3.1 技术选型

| 层级 | 技术选型 | 说明 |
|-----|---------|------|
| **前端** | Vue 3.4+ / Ant Design Vue 4.x / Pinia / Vite | 企业级 UI，B/S 分离架构 |
| **后端（Windows）** | C# .NET 8 / ASP.NET Core | Windows 原生，CNG/TPM 集成最优 |
| **后端（Linux ARM）** | Java Spring Boot 3.2+ | 跨平台，ARM 生态成熟 |
| **数据库** | SQLite（结构化）+ JSONL（哈希链） | 轻量，无外部依赖 |
| **哈希算法** | SHA-256（国际）/ SM3（国密） | 哈希链计算 |
| **签名算法** | ECDSA P-256 / SM2 | 电子签名 |
| **密钥存储** | Windows CNG + TPM 2.0（Windows）/ PKCS#11（Java） | 硬件级密钥保护 |
| **WORM 存储** | VSS 影子复制（在线）+ 蓝光 WORM（离线） | FDA 合规存证 |
| **部署方式** | Windows Service（Windows）/ Docker（Linux ARM） | 后台服务，常驻运行 |
| **前端构建** | Vite 5.x | 快速构建热更新 |

### 3.2 技术决策

| 决策项 | 结论 | 理由 |
|-------|------|------|
| **后端语言（Windows）** | C# .NET 8 | TPM/CNG 原生集成、内存占用低、启动快 |
| **后端语言（Linux ARM）** | Java Spring Boot | 跨平台兼容性好 |
| **账户体系** | 软件自建账户（独立于 OS） | FDA 审计透明、运维简单、与 TPM 密钥绑定清晰 |
| **密码算法** | Argon2id（密码 hashing） | FDA §11.100 防暴力破解，比 PBKDF2 更强 |
| **WORM 策略** | 分层：VSS 在线 + 蓝光离线归档 | 在线满足日常核查，离线满足长期归档 |

## 4. 功能模块（M1-M6）

### 4.1 模块总览

| 模块编号 | 模块名称 | 对应 FDA 合规条款 | 核心职责 |
|---------|---------|-----------------|---------|
| **M1** | 哈希链引擎 | Part 11.10(a) | 检测记录的不可篡改存储与实时完整性验证 |
| **M2** | 审计追踪系统 | Part 11.10(e) | 全部操作行为的时序记录与防篡改存储 |
| **M3** | 数字签名服务 | Part 11.50/11.200 | 检测结果的电子签名与签名验证 |
| **M4** 安全启动管理器 | Part 11.10(g) | 启动链验证，确保运行环境的可信性 |
| **M5** | 密钥与身份管理 | Part 11.100/11.300 | 用户身份管理、签名密钥的安全存储 |
| **M6** | 合规报告导出 | Part 11.10(k) | 审计报告、签名证书、完整性证明的导出 |

### 4.2 M1：哈希链引擎

#### 功能描述
每次检测记录新增时，自动将记录内容与前一块哈希合并计算 SHA-256，形成链式结构。Genesis Block 使用设备唯一标识（主板序列号 + TPM  endorsement key）生成，确保每台设备哈希链独立。

#### 核心功能

| 功能编号 | 功能名称 | 说明 |
|---------|---------|------|
| M1-001 | Genesis Block 生成 | 设备首次启动时生成创世块，记录设备唯一标识 |
| M1-002 | 哈希链追加 | 检测记录提交时自动计算并追加哈希块 |
| M1-003 | Merkle 树批量计算 | 高频检测数据批量打包，计算 Merkle Root |
| M1-004 | 根哈希锚定 | 定期（每小时/每日）将根哈希写入 WORM 存储 |
| M1-005 | 快速验链 | 仅验证最新根哈希与 WORM 锚定值一致性 |
| M1-006 | 全量验链 | 首次启动或争议场景下全量遍历验证 |
| M1-007 | 篡改告警 | 验链失败触发四级告警（INFO/WARN/CRITICAL/BREACH） |

#### 数据结构

```json
{
  "block_id": "BLK-20260701-001",
  "previous_hash": "a3f8c2d1e9b7...",
  "timestamp": 1751361600123,
  "data_hash": "sha256(检测记录内容)",
  "merkle_root": "可选，批量场景下",
  "signature": "私钥签名结果"
}
```

### 4.3 M2：审计追踪系统

#### 功能描述
记录系统中所有关键操作，覆盖用户登录、数据操作、配置变更、签名行为。审计日志独立存储，与业务数据物理隔离，追加写入模式，任意账号（含超级管理员）无法编辑、删除、覆盖。

#### 核心功能

| 功能编号 | 功能名称 | 说明 |
|---------|---------|------|
| M2-001 | 全链路日志记录 | 用户ID/终端IP/毫秒时间戳/操作类型/操作对象/变更前后/操作原因 |
| M2-002 | CRITICAL 日志即时 WORM | 关键操作日志实时写入 WORM 存储 + 哈希链 |
| M2-003 | NORMAL 日志批量 WORM | 普通操作日志按日打包，写入 WORM + 哈希链 |
| M2-004 | 日志查询与筛选 | 支持多维筛选项（时间/用户/类型/关键词） |
| M2-005 | 日志详情追溯 | 点击任意日志项，展示完整操作上下文 |
| M2-006 | 审计数据包导出 | 一键导出指定时间范围的完整审计包（PDF/CSV） |
| M2-007 | 留存周期管控 | 法定留存期内日志不可删除，到期销毁需双人审批 |

### 4.4 M3：数字签名服务

#### 功能描述
检测报告审核、批准、作废等关键操作须执行电子签名。签名强制绑定三层信息：签名人唯一ID、毫秒级精确时间戳、签署意图（初审/复核/批准/作废）。签名后记录不可篡改。

#### 核心功能

| 功能编号 | 功能名称 | 说明 |
|---------|---------|------|
| M3-001 | 电子签名签署 | 选择签署意图，输入签名密码，完成签名 |
| M3-002 | 双因素核验 | 签名时强制触发第二因素（UKey 插拔 / OTP 动态码） |
| M3-003 | 签名语义绑定 | 签名 = user_id + 毫秒时间戳 + 签署意图 + 数据哈希 + 私钥签名 |
| M3-004 | 签名验证 | 支持独立验证签名有效性（不依赖原系统） |
| M3-005 | 签名凭证展示 | 展示签名完整凭证链（可截图打印） |
| M3-006 | 签名不可剥离 | 签名与文件二元绑定，导出文件内嵌签名信息 |
| M3-007 | 签名作废 | 签名作废须双人审批，旧签名标记作废状态不可删除 |

#### 签名数据结构

```
签名凭证 = {
  signer_id:       "USR-10001",           // 签名人唯一ID
  signer_name:     "张三",
  intent:          "APPROVE",             // 签署意图：PREVIEW/REVIEW/APPROVE/VOID
  timestamp_ms:    1751361600123,         // 毫秒级时间戳
  data_hash:       "sha256(报告内容)",
```

## 7. WORM 存储与哈希链

### 7.1 分层 WORM 架构

```
┌──────────────────────────────────────────────────────────────┐
│  在线层：NTFS 只读属性 + VSS 影子副本 + 哈希链实时固化        │
│  → 日常合规审查，篡改实时检测                                 │
├──────────────────────────────────────────────────────────────┤
│  近线层：VSS 自动快照保留策略（每日/每周）                   │
│  → 篡改发生后可追溯历史版本                                 │
├──────────────────────────────────────────────────────────────┤
│  离线层：蓝光 WORM 光盘 / LTO WORM 磁带 + 介质台账           │
│  → FDA 法定长期归档（2年+）                                 │
└──────────────────────────────────────────────────────────────┘
```

### 7.2 Windows 平台 WORM 实现

| 层级 | 实现方案 | 说明 |
|-----|---------|------|
| **在线只读** | NTFS `deny write` ACL | 审计目录禁止写入权限（含 Administrator） |
| **影子副本** | Windows VSS | 自动创建只读快照，篡改后可恢复 |
| **哈希链** | JSONL 追加写入 | 每条日志实时追加，已写入内容不可修改 |
| **离线归档** | 蓝光刻录 + WORM 模式 | 月度全量归档，介质台账记录 |

### 7.3 哈希链设计

**Genesis Block 生成**：
```
设备唯一标识 = SHA256(主板序列号 + TPM Endorsement Key + 出厂时间)
Genesis Hash = SHA256("GENESIS" + 设备唯一标识 + ISO时间戳)
```

**哈希块追加**：
```
Block[N].hash = SHA256(Block[N].data_hash + Block[N-1].hash + Block[N].timestamp)
```

### 7.4 篡改检测与告警

| 告警等级 | 触发条件 | 响应动作 |
|---------|---------|---------|
| **BREACH** | 哈希链断裂/审计日志被删除 | 立即告警，阻止系统正常运行，需管理员介入 |
| **CRITICAL** | 哈希校验失败/越权访问被拦截 | 即时告警通知管理员，生成审计记录 |
| **WARN** | 介质容量不足/校验临近到期 | 提醒管理员处理 |
| **INFO** | 正常验链通过/备份完成 | 仅记录，不触发通知 |

## 8. 电子签名规范

### 8.1 FDA Part 11 电子签名要求

| FDA 条款 | 要求 | SIDU-LATS 实现 |
|---------|------|---------------|
| §11.50 | 签名须包含身份与签署意图声明 | 签名凭证含 user_id + intent + timestamp |
| §11.100 | 签名由标识符+认证码组成 | user_id + Argon2id 密码 hash |
| §11.200 | 电子签名不可转移、不可伪造 | 私钥存储 TPM/CNG，签名需密码+第二因素 |
| §11.300 | 签名认证码须保密 | TPM 硬件保护，私钥不可导出 |

### 8.2 签名流程

```
1. 操作员提交报告审核请求
2. 系统展示待签名的报告内容摘要
3. 操作员选择签署意图（初审 / 复核 / 批准 / 作废）
4. 系统要求第二因素核验（UKey 插拔 / OTP 输入）
5. 第二因素验证通过后，使用 TPM/CNG 中存储的私钥对以下数据签名：
   - 签名数据 = user_id + intent + timestamp_ms + SHA256(报告内容) + nonce
6. 签名结果写入审计日志 + 哈希链
7. 签名凭证展示（可截图/打印）
```

### 8.3 防重放攻击

- 每次签名生成随机 nonce（32字节）
- 时间戳设置合理窗口（签名时间与服务器时间偏差 ≤ 5分钟）
- nonce 使用一次后标记废弃
