## 一、产品概述

### 1.1 产品定义

SIDU-LATS Phase 2 是一款面向 FDA 21 CFR Part 11 合规的**日志审计存证系统**，部署于 Windows 工控机，为 NTA 生物检测设备提供电子签名、审计追踪、WORM 不可变存储等合规能力。

### 1.2 技术栈

| 层级 | 技术选型 |
|-----|---------|
| 前端 | Vue 3 + Ant Design Vue 4 + Pinia + Vite |
| 后端（Windows） | C# .NET 8（ASP.NET Core） |
| 数据库 | SQLite + JSONL 哈希链 |
| WORM | NTFS ACL 只读 + VSS 影子复制（后续升级） |
| 账户体系 | 软件自建账户，独立于 OS |
| 电子签名 | 密码验证（MFA 作为后续升级） |

### 2.2 Scope Out（MVP 明确排除）

以下功能在 MVP 阶段不实现，作为后续迭代补充：

| 功能 | 原因 | 计划迭代 |
|-----|------|---------|
| 双因素认证（MFA/UKey/OTP） | MVP 阶段仅实现密码验证，简化复杂度 | v1.1 |
| VSS 影子复制 | MVP 使用 NTFS ACL 只读替代，简化实现 | v1.2 |
| WORM 蓝光离线归档 | MVP 使用本地 WORM 目录，介质管理作为后续 | v1.2 |
| 安全启动管理器（M4） | 需要 TPM 硬件支持，MVP 阶段跳过 | v1.3 |
| Merkle 树批量计算 | MVP 使用单条哈希链简化实现 | v1.2 |
| SM2 国密算法 | MVP 使用国际 ECDSA P-256 | v2.0（国内药监场景） |
| IQ/OQ/PQ 验证文档 | MVP 功能完成后另行输出 | v1.x |
| 异地灾备 | MVP 单机部署，暂不需要 | v2.0 |
| LDAP/AD 集成 | MVP 纯软件账户，无需集成 | v1.x |

### 2.3 MVP 合规覆盖率

| FDA Part 11 条款 | 条款要求 | MVP 覆盖情况 |
|----------------|---------|------------|
| §11.10(a) 电子记录完整性 | 防止篡改/未授权修改 | ✅ 覆盖（哈希链+NTFS只读） |
| §11.10(b) 审计追踪 | 记录创建/修改/删除操作 | ✅ 覆盖（全链路日志） |
| §11.10(c) 操作人员身份识别 | 身份识别与操作绑定 | ✅ 覆盖（软件账户体系） |
| §11.10(d) 权限分离 | 不同职责权限分离 | ✅ 覆盖（四角色三权分立） |
| §11.10(e) 电子签名控制 | 签名与记录关联 | ✅ 覆盖（签名语义绑定） |
| §11.10(g) 记录备份 | 定期备份记录 | ⚠️ 部分覆盖（本地备份，暂无离线归档） |
| §11.10(k) 电子记录导出 | 能导出可读格式 | ⚠️ 部分覆盖（PDF导出，后续完善） |
| §11.50 签名声明 | 身份+意图明确声明 | ✅ 覆盖 |
| §11.100 签名组件 | 标识符+认证码 | ✅ 覆盖（user_id+密码） |
| §11.200 签名绑定 | 签名不可转移/伪造 | ⚠️ 部分覆盖（无MFA，签名密钥 TPM 保护待升级） |
| §11.300 认证码保密 | 认证码保密 | ✅ 覆盖（密码加密存储） |

**覆盖率评估**：MVP 可覆盖 FDA Part 11 核心条款约 **80%**，剩余条款在后续迭代中逐步完善。

## 三、用户故事

### US-001：系统管理员初始化系统

```
作为：系统管理员
背景：系统首次部署，需要初始化超级管理员账号并完成基础配置
目标：创建系统首个管理员账号，设置密码策略，配置基础参数

场景：
  1. 系统管理员访问系统初始化页面
  2. 输入超级管理员的用户名/密码/确认密码
  3. 系统验证密码符合复杂度要求（≥8字符，含大小写+数字）
  4. 系统创建管理员账号，自动分配 SYS_ADMIN 角色
  5. 系统生成 Genesis Block，记录设备唯一标识
  6. 系统初始化审计日志表和哈希链
  7. 系统进入正常运营状态

验收标准：
  - [ ] 超级管理员密码不符合复杂度时系统拒绝创建并提示
  - [ ] 超级管理员账号创建后无法被删除（保护机制）
  - [ ] Genesis Block 包含主板序列号/TPM endorsement key/时间戳
  - [ ] 系统初始化过程产生 AUDIT_LOGIN 类型 CRITICAL 日志
  - [ ] 其他管理员无法执行系统初始化
```

### US-002：QA 审计员审查审计日志

```
作为：QA 审计员
背景：药监部门即将进行现场核查，需要对近期审计日志进行全面审查
目标：快速定位关键操作，验证日志完整性，出具合规审查报告

场景：
  1. QA 审计员使用账号登录系统
  2. 进入审计日志页面，设置时间范围（近3个月）和事件类型（CRITICAL）
  3. 系统展示筛选后的日志列表，CRITICAL 日志红色高亮
  4. 点击某条日志，右侧抽屉展示完整详情（用户/时间戳/IP/变更前后内容）
  5. 执行哈希链完整性校验，系统输出 PASS
  6. 导出指定时间范围的合规数据包（PDF格式）
  7. 系统记录导出操作到审计日志（REPORT_EXPORT）

验收标准：
  - [ ] QA 审计员仅能查看日志，无法修改任何数据
  - [ ] CRITICAL 日志与 NORMAL 日志有明确视觉区分
  - [ ] 哈希链校验失败时系统展示具体断裂位置
  - [ ] 导出数据包包含原始日志 + 哈希校验文件
  - [ ] 导出操作本身被记录为 CRITICAL 日志
```

### US-003：审核人员对检测报告进行电子签名

```
作为：审核人员
背景：某批次 NTA 检测已完成，需要审核人员初审 + 复核后最终批准
目标：完成两级审批电子签名，形成合规的签名记录供药监核查

场景（初审）：
  1. 审核人员登录系统，进入待审批列表
  2. 选择待初审的检测报告，查看报告内容摘要
  3. 点击"初审签名"，系统展示签名确认面板
  4. 选择签署意图：初审（PREVIEW）
  5. 输入账号密码进行身份核验
  6. 系统生成签名凭证（user_id + PREVIEW + timestamp_ms + 报告哈希 + 签名值）
  7. 系统将签名凭证写入审计日志（ELEC_SIGN）
  8. 签名凭证面板展示，可截图或打印

场景（复核/批准）：
  9. 另一名审核人员登录（单人不能完成两级审批）
  10. 进入同一报告，选择签署意图：批准（APPROVE）
  11. 输入密码确认
  12. 系统生成最终批准签名，报告状态更新为 APPROVED
  13. 全流程审计日志留存，包含初审+复核两个签名

验收标准：
  - [ ] 初审和复核不能由同一账号完成
  - [ ] 签名强制绑定 user_id + 毫秒时间戳 + 签署意图
  - [ ] 签名数据中包含报告内容的 SHA-256 哈希值
  - [ ] 签名后报告内容任何修改均导致签名验证失败
  - [ ] 签名凭证可截图打印，无需登录系统可验证签名有效性
  - [ ] 密码错误3次后账号锁定30分钟
```

### US-004：系统自动生成哈希链并完成验链

```
作为：系统（自动）/ QA 审计员（手动触发）
背景：检测记录持续写入，需要哈希链实时更新；定期验链确保数据完整性
目标：哈希链准确反映所有新增记录，被篡改时能立即发现

场景（实时追加）：
  1. NTA 设备上传一条新的检测记录
  2. 系统计算记录内容 SHA-256 哈希
  3. 系统获取当前哈希链的最新块哈希
  4. 系统计算新块哈希 = SHA256(记录哈希 + 前一块哈希 + 时间戳)
  5. 新块写入哈希链，sequence 自增
  6. 若当前时间达到整点，执行根哈希锚定（写入备份文件）

场景（快速验链）：
  7. QA 审计员点击"验链"按钮
  8. 系统读取当前最新块，提取 self_hash
  9. 系统对比根哈希备份文件中的锚定值
  10. 一致则输出 PASS，不一致则触发 BREACH 告警

场景（全量验链）：
  11. QA 审计员选择"全量验链"
  12. 系统从 Genesis Block 开始逐块验证 self_hash 连续性
  13. 任何一块验证失败则停止，输出断裂位置
  14. 全量通过输出 PASS，全量耗时作为性能指标记录

验收标准：
  - [ ] 每条新记录追加哈希链耗时 < 100ms
  - [ ] Genesis Block 使用设备唯一标识生成，无法伪造
  - [ ] 验链结果包含：通过/断裂位置/断裂时间
  - [ ] 验链失败触发告警，通知系统管理员
  - [ ] 哈希链文件写入权限仅限 SYSTEM + 管理员
```

### US-005：系统自动执行 WORM 归档

```
作为：系统（自动执行）
背景：CRITICAL 审计日志需要实时写入 WORM 存储；NORMAL 日志按日打包归档
目标：审计日志不可篡改，留存满足 FDA 要求（≥2年）

场景（CRITICAL 日志实时写入）：
  1. 用户执行关键操作（登录/签名/权限变更）
  2. 系统生成 CRITICAL 审计日志，timestamp 精确到毫秒
  3. 系统立即追加写入 audit_log.worm.jsonl 文件（append-only）
  4. 同时写入 SQLite audit_log 表
  5. 审计目录设置 NTFS deny write ACL，审计日志文件不可修改

场景（NORMAL 日志日包归档）：
  6. 每日 23:00，系统检查当日 NORMAL 日志
  7. 将当日 NORMAL 日志打包计算 SHA-256 哈希
  8. 将打包文件写入 archive/{年份}/{月份}/ 目录
  9. 更新哈希链（批量 Merkle 树，后续升级）
  10. 生成归档记录，写入介质台账

场景（WORM 存储保护验证）：
  11. 任意尝试修改/删除 audit_log.worm.jsonl 的操作被系统拒绝
  12. 即使使用 Administrator 账号也无法绕过
  13. 异常操作产生 CRITICAL 审计日志（系统自身行为）

验收标准：
  - [ ] CRITICAL 日志写入 WORM 存储延迟 < 1秒
  - [ ] 审计日志目录 NTFS ACL 仅允许 SYSTEM 写入
  - [ ] 任意用户（含 Administrator）尝试修改日志文件返回拒绝
  - [ ] 日志留存周期可配置，默认 ≥ 2年
  - [ ] 留存期内日志无法被物理删除
```

## 七、风险评估

| 风险 | 概率 | 影响 | 缓解措施 |
|-----|------|-----|---------|
| NTFS ACL 配置被绕过 | 低 | 高 | 定期手动抽检 + 告警监控 |
| 签名私钥泄露 | 低 | 极高 | 优先使用 TPM 存储密钥，软件方案加密保护 |
| 验链性能不足（数据量大时） | 中 | 中 | Merkle 树批量优化（v1.2） |
| FDA 核查员对 MVP 合规性存疑 | 中 | 高 | 提前沟通 MVP 范围，明确后续迭代计划 |
| .NET Runtime 工控机兼容性 | 低 | 中 | 选择 LTS 版本，部署前做兼容性测试 |
| 密钥丢失（无 TPM 场景） | 低 | 极高 | 密钥备份机制（加密导出，分散存储） |
