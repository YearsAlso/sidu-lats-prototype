# Design: 移除报告导出页的"导出需审批"叙事

## 设计决策

### D1:仅改 UI 文案/DOM,不改 JS 逻辑

JS 逻辑层(`generateReport`/`downloadReport`/`verifyReport`/`downloadSample` 直接触发 alert)本身不违反 PRD——PRD 本就不要求导出审批。本次只移除 UI 文案层凭空引入的审批叙事,不动 JS 函数,降低改动半径与回退成本。

### D2:替换而非删除提示卡

原"报告导出需审批"warning 卡占位重要(提示用户导出合规约束)。直接删除会留下视觉空白且丢失合规提示价值。替换为"导出合规约束"卡,文案对齐 PRD 有据可查的三条约束:
- M6-004:导出包内嵌哈希防篡改
- M6-005:仅 QA 审计员/系统管理员可导出
- M6-006:导出操作写审计日志且不可删除
- Part 11 §11.10(k):电子记录精确复制,离线可核验

### D3:删除"待审批导出申请"块

`APR-20260701-006(Q2审计报告)`审批中单据是 PRD 不存在的编造数据,整块删除(无替换)。该块是独立 `<div>`,删除不影响相邻的"合规证明"快捷卡(purple 底,L190-194)与父容器布局。

### D4:warning 配色保持

提示卡保留 antd warning 配色(`#fffbe6` 底 / `#ffe58f` 描边 / `#d48806` 标题字),语义上"合规约束提醒"与 warning 色系契合,无需改色。

### D5:两页文案完全一致

admin 与 auditor 两个导出页的偏差区域当前完全一致,修改后须保持一致(同一份文案),避免角色间合规说明出现差异引发新的不一致。

### D6:不动相邻 purple 快捷卡

"合规证明"快捷卡(`#f9f0ff` 底 / `#d3adf7` 描边,purple 配色)虽不在 antd 标准色板内,但属本次范围外,保留不动(避免改动半径扩大)。如后续需统一,另立 change。

## 不做

- 不给导出按钮加审批前置 Modal(反向加戏,违反 M6-005)
- 不改导出历史记录区、表格、预览 Modal、tab 切换
- 不改侧栏菜单与 sessionStorage 鉴权流
- 不改需求文档(已冻结)
