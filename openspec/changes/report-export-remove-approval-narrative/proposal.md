# Proposal: 移除报告导出页的"导出需审批"叙事

## Why

PM 对照冻结 PRD V3.0 审查确认:**报告导出不需要他人审批**。需求依据:

- §4.7 M6-005:导出权限控制仅规定"谁能导"(QA 审计员/系统管理员),未规定"需谁批"
- §4.7 M6-006:合规抓手是事后审计留痕(不可删除),非事前审批
- §6.2 权限矩阵:合规报告导出为单角色动作,无"发起人+审批人"两段式
- §4.2 M2-007 / §4.4 M3-007:PRD 中仅有的两处双人审批分别作用于**日志销毁**与**签名作废**,不作用于报告导出
- §8 ESig 规范:电子签名强制场景仅"报告审核/批准/作废",导出不在 ESig 清单内
- FDA 21 CFR Part 11 §11.10(k):导出属"电子记录精确复制 + 可核验 + 留审计日志",不要求事前签名审批

当前 `prototype/admin/report-export.html` 与 `prototype/auditor/report-export.html` 的 JS 逻辑层(按钮 onclick 直接触发下载)**本身不违反 PRD**;但 UI 文案层凭空引入了 PRD 不存在的"导出需审批 / 审批中心 / 待审批申请单"叙事:

1. warning 提示卡:"导出超1000条记录或含签名的报告,需由审计员或管理员在审批中心提交申请,经审批通过后方可下载"(admin L198-199 / auditor L197-198)
2. "待审批导出申请"列表项 `APR-20260701-006(Q2审计报告)` 状态"审批中"(admin L201-207 / auditor L200-207)

该叙事会误导演示受众认为 SIDU-LATS 把导出做成了审批制,与 §6.2 权限矩阵("QA 审计员可独立导出")直接矛盾,审计方对照矩阵会判定权限模型自相矛盾。必须移除,改为 PRD 有据可查的合规说明。

## What Changes

- **替换** 两个导出页的"报告导出需审批"warning 提示卡 → "导出合规约束"提示卡(antd warning 配色不变),文案对齐 M6-004(防篡改哈希)/ M6-005(角色权限)/ M6-006(审计留痕不可删)/ Part 11 §11.10(k)(精确复制)
- **删除** 两个导出页的"待审批导出申请"块(`APR-20260701-006` 审批中单据)整块 DOM
- **不改**:JS 函数(`generateReport`/`downloadReport`/`verifyReport`/`downloadSample`)、表格、预览 Modal、侧栏、tab 切换、sessionStorage 鉴权流、导出历史记录区
- **不改**:需求文档(已冻结;本次是原型对齐需求,非需求变更)

## Capabilities

### Modified Capabilities
- `role-admin`: admin 报告导出页移除审批叙事,提示卡对齐 M6 合规约束
- `role-auditor`: auditor 报告导出页移除审批叙事,提示卡对齐 M6 合规约束

(无 New Capabilities)

## Impact

- **代码**:2 文件(`prototype/admin/report-export.html`、`prototype/auditor/report-export.html`),每文件 ~12-15 行(替换 1 个提示卡 + 删除 1 个审批申请块),合计 ≤30 行
- **构建配置**:无改动(页面入口不变)
- **文档**:无改动
- **鉴权流**:无改动
- **风险**:低,纯文案/DOM 替换,单 commit 可回滚;需确保两页文案一致;不得误删相邻的"合规证明"快捷卡片(purple 底,虽非本次范围,保留不动)
- **需求文档**:不改(已冻结)

## 验收标准

1. 两个导出页搜索"审批""审批中心""APR-20260701"均无残留
2. 提示卡文案包含 M6-004/005/006 与 Part 11 §11.10(k) 语义
3. 提示卡 antd warning 配色保持(`#fffbe6`/`#ffe58f`/`#d48808`)
4. JS 函数、表格、tab、侧栏、sessionStorage 流未受影响
5. 双击打开两页无控制台报错,导出按钮仍可点击触发原有 alert
